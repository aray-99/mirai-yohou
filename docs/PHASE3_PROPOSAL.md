## 1. 現在地の客観評価 — このプロジェクトはどこまで来たか

### 1.1 検証の階段(見取り図)

このプロジェクトは「社会を物理系のように数式で書き、その数式が本当に予測力を
持つかを段階的に検証する」という構造を持つ。検証の階段は 3 段あり、現在
**2 段目までを登り切った**状態にある。

```
段階 1: モデルが数学的に正しく動くか(M0〜M4、1日で完了)
   13次元 SDE + Hawkes ジャンプ + EnKF 同化を Julia で実装。
   安定国/変動国のレジームが設計どおり分岐し、ファンチャートが出る。
        ↓ PASS
段階 2: モデルが「答えを知っている世界」で真値を復元できるか(M5〜M10)
   ├─ 双子実験(M5): 自分で生成した合成データから状態・パラメータを
   │   復元。E1b PASS、E1a 6/8(#0019)         … 2026-07-07
   └─ 実データ検証(M8〜M10): 日本・タイの実データ(世銀・ACLED・
       V-Dem・SWIID・WVS)で walk-forward ヒンドキャスト。
       「その年までのデータだけで翌年以降を予測」を 6〜8 起点で反復し、
       90% 予報区間の被覆率で判定。JPN 0.885 / THA 0.851 — 全主要基準
       PASS(#0069)                            … 2026-07-14 ← いまここ
        ↓
段階 3: 未来を予報し、それを見せる(未着手)
   真の将来予報(2026年起点)+ 予報円ダッシュボード = ユーザーのゴール像
```

### 1.2 到達点の定量サマリ

| 項目 | 値 | 意味 |
|---|---|---|
| 90% 予報区間の被覆率(JPN) | **0.885**(54/61) | 要求帯 [0.80, 0.98] 内。自由ラン 0.656 を大きく改善 |
| 同(THA) | **0.851**(40/47) | 同上。自由ラン 0.532 から +0.32 |
| 紛争イベント数の予測対数尤度 | JPN −48 vs 自由 −93 / THA −434 vs −504 | 同化が「事件の起きやすさ」の予測を明確に改善 |
| RMSE 改善変数 | JPN 7/9、THA 8/9 | ほぼ全変数で自由ランに勝つ |
| テスト | 2510 本 green | 座標変換から同化まで全層に回帰テスト |
| 開発規模 | コミット 116、DECISIONS 69 エントリ、src 13 モジュール | 全決定が追跡可能 |

**信頼できる範囲の率直な評価**: 「1〜6 年先の予報区間が、日本とタイの直近
10〜15 年について、約束した確率で当たる」ことは実データで実証済み。逆に、
(a) 2 カ国以外では未検証、(b) 検証はヒンドキャスト(過去の再予測)であり
真の将来予報の運用実績はゼロ、(c) 弱い変数が残る(政府信頼 tau は両国で
観測が疎すぎて被覆 0/1、THA の技術代理変数 T_proxy 2/4、JPN の phi 4/8)。

### 1.3 なぜ M1〜M4 は 1 日で進み、Phase 2 は泥臭かったのか

M1〜M4 は「自分で設計した世界」の中の話であり、数学とコードの正しさだけが
問われた。Phase 2 で相手が実データに変わった瞬間、問題の性質が変わった:

- **同定性**: 年 1 点の観測 × 十数個のパラメータでは、較正(EKI)が「どの
  パラメータのせいか」を区別できない(#0051 の THA 不合格、#0061 の
  lam0/eta_g ドリフト)。M10 の事前分布アンカリングでようやく解決
- **データの疎さと癖**: ジニ係数は 2 年遅れ、WVS は 5 年おき、タイの ACLED は
  深南部紛争が死者の 85% を占め国政分析には除外必須(#0026/#0030)
- **過分散**: 素のポアソン尤度では粒子重みが 1 点に崩壊(#0033)

この 2 週間の「終わりの見えなさ」は、**モデルの予測性能を凍結基準で愚直に
検証し、不合格のたびに原因を診断して是正した**ことの直接の結果である。
基準を動かさずに 3 マイルストーン(M8 部分達成 → M9 部分達成 → M10 合格)を
要したが、その過程が DECISIONS #0040〜#0069 として全部残っており、
「なぜこのモデルを信じてよいか」の証拠になっている。これは生半可ではない。

## 2. ゴールとのギャップ分析

ユーザーのゴール像: **「数年後〜数十年後の未来の予報円が、モダンな
ダッシュボードとして可視化された状態」**。現在地との差は 3 つに分解できる。

### ギャップ① ヒンドキャスト → 真の予報

現在のパイプラインは過去の再予測専用。将来予報には:
- **ACLED エンバーゴ**: イベントデータは直近 12 ヶ月非公開 → 同化の最終年は
  常に約 1 年遅れる。予報起点は「データが揃う最新年」(現時点で 2024〜25 年頃)
- 世銀 GDP・SWIID も 1〜2 年遅れ。つまり「2026 年からの予報」は正確には
  「2024 年頃の状態推定から出発する 2〜30 年予報」になる — これは天気予報の
  「初期値がやや古い」状況に相当し、設計上は問題なく扱える

### ギャップ② 静的 PNG → ダッシュボード

可視化資産は現状 Plots.jl の静的 PNG(M3 ファンチャート)のみ。方式選定は
本提案 §4.2 参照(結論: 来歴埋め込み型の自己完結 HTML + GitHub Pages)。

### ギャップ③ 2 カ国 → 横展開

現在の M8〜M10 スクリプトは JPN/THA の知見がハードコード気味(較正設定、
ACLED 除外規則等)。「国コードを渡せば全工程が流れる」一般化がされていない。

### 科学的誠実さの線引き(重要)

被覆率が実証されているのは **6 年先まで**。「数十年後の予報円」は検証の外側で
あり、当たる保証を主張できない。ダッシュボードでは検証済みホライズン
(〜6 年)と外挿領域(それ以降)を視覚的に区別して表示する — 「予報」でなく
「このモデルが描く不確実性の広がり」として提示する。これは弱点の隠蔽ではなく
気象庁の予報円が期間で信頼度を変えるのと同じ流儀である。

## 3. 開発体制: Issue ベース開発への転換(ユーザー提案採用)

リポジトリは public(github.com/aray-99/mirai-yohou)で Issues 有効。
友人(Fable 枠余剰者)はフォーク + PR で collaborator 権限なしでも参加できる。

- **Issue = 委譲プロンプトの永続版**。現行のサブエージェント指示の流儀
  (目的・対象ファイル・完了条件・報告形式を自己完結で書く)をそのまま
  Issue 書式にする。Sonnet への Agent 指示も友人への依頼も
  「Issue #N を解決せよ」に一本化
- **ラベル**: マイルストーン(`M11` `M12` `M13`)× 必要能力(`fable-required` /
  `sonnet-ok` / `good-first-issue`)× 状態
- **Fable に残す役割**: マイルストーンの Issue 分解、PR レビュー、受入判定
  (凍結基準の番人 — 外部 PR の自己申告 green を鵜呑みにせずローカルで検証)、
  DECISIONS 起草、科学的判断。マージ規約(feature → develop `--no-ff`)不変
- **環境要件の明記**: ACLED 認証情報は共有不可。データ取得系 Issue は
  「キャッシュ済み CSV で作業可能」か「要 ACLED アカウント」かを明記。
  walk-forward 検証(約 2 時間/国)は実行環境と枠を消費する旨も Issue に書く
- 体制導入自体を M11 の最初の Issue 群として起票(.github/ISSUE_TEMPLATE 整備込み)

## 4. ロードマップ — 横展開可能な全体像の完成を最短経路に

方針: 課題を幅広く積むのではなく、**データ → 較正 → 検証 → 予報 → 可視化の
エンドツーエンドを先に閉じる**。深化系の課題は全体像完成後のバックログとする。

### M11: 予報パイプラインの一般化 + 真の将来予報

- **目的**: JPN/THA 固有の手続きに散らばった工程を「国コードを引数に取る
  単一パイプライン」に再編し、その最初の適用として JPN/THA の最新データ起点
  (2024〜25 年頃)の将来予報アンサンブル(〜30 年先)を JSON で出力する
- **完了条件(素案)**: (1) `forecast_pipeline.jl COUNTRY` 一発でデータ取得→
  観測整形→較正→walk-forward 自己検証→将来予報 JSON が生成される、
  (2) JPN/THA で再実行した walk-forward が M10 と同水準(被覆帯内)、
  (3) 予報 JSON に来歴(コミット・シード・データ取得日)サイドカー
- **作業量見込み**: 設計 + Issue 分解が Fable 1 セッション、実装は Sonnet/友人
  委譲で Issue 5〜8 本、較正・検証ランが計 4〜6 時間(機械時間)
- **リスク/律速**: リファクタで M10 合格構成を壊さないこと(凍結設定
  M8_frozen_config.toml と #0062 プロトコルの互換維持が受入の要)。
  長期予報での数値ガード(#0032 の tame+floor)が 30 年ホライズンでも安定かは
  スモークで先に確認する

### M12: 予報円ダッシュボード

- **目的**: M11 の予報 JSON を読む自己完結 HTML ダッシュボード(依存 CDN なし、
  データ・来歴埋め込み)。国切替、13 変数 + 診断量のファンチャート(予報円)、
  検証済みホライズンと外挿領域の視覚的区別、来歴表示。GitHub Pages で公開
- **方式選定の理由**(Fable 判断): 静的 HTML + JSON 埋め込みは (a) 来歴主義
  (SPEC §0.5.6「どのコミット・シードの予報円か遡れる」)と完全整合 —
  ダッシュボード自体が git 管理され来歴を内蔵する、(b) Julia スクリプト 1 本で
  決定的に再生成可能、(c) 開発中は Claude Artifact でプレビューでき(CSP 制約
  = 自己完結要件と同一)、完成品は GitHub Pages で常時閲覧可能。
  Makie.jl は静的図の高品質化には良いがインタラクティブ配信に不向き、
  サーバ型 Web アプリは再現性・保守の思想に反するため不採用
- **完了条件(素案)**: JPN/THA の予報円が国切替つきで表示され、来歴
  (コミット SHA・シード・データ取得日)が UI 上で確認でき、GitHub Pages の
  URL でユーザーが閲覧できる
- **作業量見込み**: デザイン仕様と受入判定が Fable、実装は Sonnet/友人委譲で
  Issue 3〜5 本。機械時間ほぼゼロ
- **リスク**: 小(検証ラン不要、凍結基準に触れない)。デザイン品質だけが論点

### M13: 横展開(3 カ国目以降)

- **目的**: 一般化パイプラインに新しい国を流し、ダッシュボードに追加する
- **前提調査**: ACLED カバレッジ × V-Dem × SWIID × 世銀の交差で候補国を選定
  (Fable が 1 Issue で調査委譲 → 選定判断)。タイの深南部除外(#0026)のような
  国固有の前処理判断が毎回発生する見込み — ここだけは Fable/ユーザーの判断が要る
- **完了条件(素案)**: 新国で walk-forward 被覆が帯内 → ダッシュボード追加。
  国ごとに 1 サイクル
- **作業量見込み**: 1 カ国あたり Issue 2〜3 本 + 検証ラン 2〜4 時間。
  パイプライン化済みならほぼ委譲可能な定型作業(`sonnet-ok` / 友人向き)
- **リスク/律速**: データ可用性が全て。ACLED は国により開始年が大きく違う。
  被覆が帯を外れた場合の是正は M10 の流儀(是正上限 2 回)を踏襲

## 5. 推奨とユーザー判断事項

**推奨**: M11 → M12 → M13 の順で着手し、まず「日本とタイの未来の予報円が
ダッシュボードで見える」状態(= ゴール像の成立)を作る。見込みは M11+M12 で
Fable 2〜3 セッション + 委譲 Issue 10 本前後。M13 で国を増やしながら、
必要になった深化課題(特に tau の EnKS)だけをバックログから引く。

**ユーザーに判断いただくこと**:
1. M11〜M13 のマイルストーン化と順序の承認(承認後、PHASE3_DESIGN として
   完了条件を凍結し DECISIONS に記録)
2. Issue ベース体制の開始承認(ラベル・テンプレートの整備、友人への公開)
3. M13 の候補国の希望があれば(なければデータ可用性調査から Fable が提案)
