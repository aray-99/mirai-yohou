# PHASE3_DESIGN — 予報の運用化と可視化(M11〜M13)

凍結: 2026-07-14(ユーザー承認、DECISIONS #0070)
前提: Phase 2 完了(M10 合格 #0069、vM10)。提案の経緯と全体評価は docs/PHASE3_PROPOSAL.md。

## §1 スコープと方針

Phase 3 の目的は、Phase 2 で予測性能を実証したモデルを
**「データ → 較正 → 検証 → 将来予報 → 可視化」のエンドツーエンドとして完成**させ、
ユーザーのゴール像(未来の予報円がモダンなダッシュボードで見える状態)を成立させること。

- 課題の幅広い積み上げより**横展開可能な全体像の完成を優先**する(ユーザー方針)。
  深化課題(tau EnKS、y 四半期化、GDP ヴィンテージ、theta_p、EKI 正則化、
  GDELT 感度、k・tauA)は §5 バックログとし、必要時に個別にマイルストーン化
- 各マイルストーンの**詳細プロトコル(検証設定・シード等)は着手時に別途
  DECISIONS で凍結**する(M8 #0045 / M9 #0052 / M10 #0062 と同じ流儀)。
  本文書はフェーズスコープと完了条件の骨子を凍結する

## §2 M11: 予報パイプラインの一般化 + 真の将来予報

目的: JPN/THA 固有の手続きに散らばった工程を「国コードを引数に取る単一
パイプライン」に再編し、最初の適用として両国の最新データ起点の将来予報
アンサンブルを JSON で出力する。

完了条件(凍結):
1. 単一コマンド(国コード引数)でデータ取得 → 観測整形 → 較正 → walk-forward
   自己検証 → 将来予報 JSON 生成が通しで実行できる
2. JPN/THA の再実行 walk-forward が M10 凍結基準(#0052)の要求帯内
   (= リファクタが合格構成を壊していないことの回帰確認。基準の変更はしない)
3. 将来予報 JSON は予報ホライズン 30 年、来歴サイドカー(コミット SHA・シード・
   データ取得日・凍結設定)付き。検証済みホライズン(6 年)の境界をメタデータに含む

注意事項:
- 凍結設定(M8_frozen_config.toml、#0062/#0063 プロトコル)との互換を維持する
- 30 年ホライズンの数値安定性(#0032 tame+floor ガードの領域)を実装前に
  スモークで確認する
- 予報起点は「全観測系列が揃う最新年」(ACLED 12ヶ月エンバーゴにより
  2024〜25 年頃)。起点の古さは予報の初期値として明示する

## §3 M12: 予報円ダッシュボード

目的: M11 の予報 JSON を読む**自己完結静的 HTML ダッシュボード**を構築し、
GitHub Pages で公開する。

方式(凍結): 外部依存(CDN・サーバ)なしの静的 HTML にデータと来歴を埋め込み、
Julia スクリプト 1 本で決定的に再生成する。選定理由は PHASE3_PROPOSAL §4.2
(来歴主義 SPEC §0.5.6 との整合、再現性、Artifact プレビュー互換)。

表示仕様の骨子(詳細デザインは着手時に確定):
- 観測照合済み 9 系列(P, w, y, g_swiid, T_proxy, phi, v, tau, p)+
  騒乱・政治危機の発生リスク(カウント予報)の予報円(ファンチャート)
- **検証済みホライズン(〜6 年)と外挿領域の視覚的区別**。外挿領域は「予報」
  でなく「不確実性の広がり + 体質診断(De レジーム等の無次元診断)」として提示
- 国切替、来歴(コミット SHA・シード・データ取得日)の UI 表示

完了条件(凍結):
1. JPN/THA の予報円が国切替つきで表示される
2. 来歴が UI で確認できる
3. GitHub Pages の URL でユーザーが閲覧できる

## §4 M13: 横展開(3カ国目以降)

目的: 一般化パイプラインに新しい国を流し、ダッシュボードに追加する。

手順(国ごとに 1 サイクル):
1. 候補国調査(ACLED × V-Dem × SWIID × 世銀のデータ交差。委譲可)→
   選定はユーザー希望を優先、なければ Fable が提案しユーザー確認
2. 国固有の前処理判断(THA 深南部除外 #0026/#0030 に相当する判断)は
   Fable が診断しユーザーに報告(DECISIONS 記録)
3. パイプライン実行 → walk-forward 被覆が要求帯内で合格 → ダッシュボード追加

完了条件(凍結): 新国の walk-forward が M10 凍結基準の要求帯内 +
ダッシュボード反映。不合格時の是正は M10 の流儀(是正上限 2 回、上限到達で
部分達成クローズ判断をユーザーへ)を踏襲。

## §6 開発体制: Issue ベース開発

- 承認済みマイルストーンを Fable が**自己完結 Issue**(目的・対象ファイル・
  完了条件・報告形式)に分解する。Sonnet への委譲も外部協力者(フォーク + PR)
  への依頼も「Issue #N を解決」に一本化
- ラベル: マイルストーン(`M11` `M12` `M13`)× 必要能力(`fable-required` /
  `sonnet-ok` / `good-first-issue`)
- Fable に残す役割: Issue 分解、PR レビュー、受入判定(外部 PR のテスト結果は
  ローカルで再検証)、DECISIONS 起草、科学的判断。マージ規約
  (feature → develop `--no-ff`、SPEC §0.5)は不変
- Issue には環境要件を明記: ACLED 認証情報は共有不可(キャッシュ済み CSV で
  作業可能か要記載)、長時間ラン(walk-forward 約 2 時間/国)の機械時間
- 体制整備(.github/ISSUE_TEMPLATE、ラベル作成)は M11 の最初の Issue 群に含める
